## 1. ARCHITECTURE OVERVIEW

### Data Flow

```
External APIs (USGS, EONET, GDACS, FEMA, FIRMS, NOAA, ReliefWeb)
        │
        ▼
┌─────────────────────────┐
│  Python Ingestion Worker │  (Railway — FastAPI + APScheduler)
│  Polls on schedule       │
│  Normalizes to schema    │
│  Writes to Postgres      │
└─────────┬───────────────┘
          │ SQL INSERT/UPDATE
          ▼
┌─────────────────────────┐
│  Neon PostgreSQL         │  (Shared database — single source of truth)
│  Events, EventUpdates,   │
│  SourceStatus            │
└─────────┬───────────────┘
          │ Prisma Client reads
          ▼
┌─────────────────────────┐
│  Next.js App (Vercel)    │
│  ├─ API Routes (/api/*)  │  ← Frontend fetches from these
│  ├─ App Router pages     │
│  ├─ 3D Globe (Three.js)  │
│  └─ Event pages (Leaflet)│
└─────────────────────────┘
```

### Deployment Topology

- **Vercel (free tier)**: Hosts the Next.js app. Serves SSR pages, API routes, and static assets. Environment variables configured in Vercel dashboard.
- **Neon (free tier)**: Serverless PostgreSQL. Single database accessed by both the Next.js app (via Prisma) and the Python worker (via psycopg2). **Critical:** The Next.js app must use Neon's pooled connection string (PgBouncer endpoint, port 6432) as its `DATABASE_URL`. Neon's free tier allows only a small number of direct connections; Vercel's serverless functions open a new connection per invocation and will exhaust the limit under any real traffic. The Python worker on Railway is a long-lived process and should use the direct (non-pooled) connection string to avoid PgBouncer transaction-mode limitations that break multi-statement transactions. Both strings are available from the Neon dashboard — expose both as env vars (see Section 8).
- **Railway ($5/mo credit)**: Runs the Python ingestion worker as a long-lived process. Dockerfile-based deployment from the `/worker` directory. Keeps the scheduler alive 24/7.

### Key Tradeoffs

1. **Two runtimes (Node + Python)**: Added deployment complexity, but Python has far superior libraries for XML/GeoRSS parsing, geospatial processing, and HTTP polling patterns. The worker is a simple standalone service — no shared code needed.
2. **Polling, not streaming**: All sources are polled on intervals. This is simpler, free-tier-compatible, and matches how these APIs actually deliver data. V2 can add SSE push from the Next.js layer.
3. **Prisma for reads, raw SQL for writes**: The Next.js app uses Prisma for type-safe reads. The Python worker uses raw SQL via psycopg2 for writes — no need for a Python ORM when we're doing bulk upserts.
4. **AI summaries generated on-demand**: Gemini Flash summaries are triggered when a user visits an event page, not during ingestion. This avoids burning API quota on events nobody views.
5. **No real-time push in MVP**: Frontend polls every 60 seconds. Adequate for the data update frequency of our sources (5-60 min intervals).

## 4. API CONTRACT

### GET /api/events

List events with optional filters.

**Query Params:**
| Param    | Type   | Default | Description |
|----------|--------|---------|-------------|
| type     | string | —       | Comma-separated DisasterType values. e.g. `EARTHQUAKE,WILDFIRE` |
| severity | string | —       | Comma-separated Severity values. e.g. `HIGH,EXTREME` |
| source   | string | —       | Comma-separated EventSource values |
| bbox     | string | —       | `minLat,minLon,maxLat,maxLon`. **Antimeridian handling required:** if `minLon > maxLon` (e.g. `30,170,60,-170` spanning the 180° line), the SQL must use `(lon >= minLon OR lon <= maxLon)` instead of the standard `(lon >= minLon AND lon <= maxLon)`. Detect this condition before building the query. |
| since    | string | —       | ISO 8601 datetime. Only events with `started_at >= since` |
| limit    | number | 500     | Max events returned. Capped at 2000. |

## 5. INGESTION PIPELINE SPEC

### Base Ingestor Pattern

Every ingestor inherits from a base class that provides:
- HTTP session with retry logic (3 retries, exponential backoff)
- Error catching: exceptions are caught, logged, and written to SourceStatus — never crash the scheduler
- Deduplication via `ON CONFLICT (source, external_id) DO UPDATE`
- Change detection: before updating, compare old vs new field values. If any tracked field changed, write an EventUpdate record. Tracked fields: title, description, severity, status, lat, lon, geometry, country, region.

### USGS Earthquake API

- **Endpoints:**
  - `https://earthquake.usgs.gov/earthquakes/feed/v1.0/summary/all_hour.geojson` (every 5 min)
  - `https://earthquake.usgs.gov/earthquakes/feed/v1.0/summary/all_day.geojson` (every 30 min)
  - `https://earthquake.usgs.gov/earthquakes/feed/v1.0/summary/2.5_week.geojson` (every 6 hours)
- **Polling interval:** 5 minutes (hourly feed), 30 minutes (daily feed), 6 hours (weekly feed)
- **Auth:** None
- **Dedup key:** `source=USGS, external_id=features[].id`
- **Field mapping:**
  | USGS Field | Schema Field |
  |------------|-------------|
  | `features[].id` | `external_id` |
  | `"USGS"` | `source` |
  | `"EARTHQUAKE"` | `type` |
  | `features[].properties.title` | `title` |
  | `features[].properties.place` | `description` |
  | magnitude-based (see below) | `severity` |
  | `features[].properties.status == "reviewed" ? "ACTIVE" : "UNKNOWN"` | `status` |
  | `features[].geometry.coordinates[1]` | `lat` |
  | `features[].geometry.coordinates[0]` | `lon` |
  | `null` | `geometry` |
  | extracted from `properties.place` (after last comma) | `country` |
  | extracted from `properties.place` (before last comma) | `region` |
  | `datetime.fromtimestamp(properties.time / 1000)` | `started_at` |
  | full feature object | `raw_data` |
- **Severity mapping from magnitude:**
  - mag < 3.0 → `LOW`
  - 3.0 <= mag < 5.0 → `MODERATE`
  - 5.0 <= mag < 7.0 → `HIGH`
  - mag >= 7.0 → `EXTREME`

### NASA EONET

- **Endpoint:** `https://eonet.gsfc.nasa.gov/api/v3/events?status=open&limit=100`
  - Optionally append `&api_key={NASA_API_KEY}` if env var set (higher rate limit)
- **Polling interval:** 15 minutes
- **Auth:** Optional NASA API key
- **Dedup key:** `source=EONET, external_id=events[].id`
- **Field mapping:**
  | EONET Field | Schema Field |
  |-------------|-------------|
  | `events[].id` | `external_id` |
  | `"EONET"` | `source` |
  | category-based mapping (see below) | `type` |
  | `events[].title` | `title` |
  | `events[].description` or null | `description` |
  | `null` (EONET has no severity) | `severity` |
  | `events[].closed ? "CLOSED" : "ACTIVE"` | `status` |
  | last geometry point `coordinates[1]` | `lat` |
  | last geometry point `coordinates[0]` | `lon` |
  | full geometry if polygon | `geometry` |
  | null (not provided) | `country` |
  | null | `region` |
  | `events[].geometry[-1].date` (most recent) | `started_at` |
- **EONET category → DisasterType mapping:**
  - `wildfires` → WILDFIRE
  - `volcanoes` → VOLCANO
  - `severeStorms` → STORM
  - `floods` → FLOOD
  - `earthquakes` → EARTHQUAKE
  - `drought` → DROUGHT
  - everything else → OTHER

### GDACS GeoRSS

- **Endpoint:** `https://www.gdacs.org/xml/rss.xml`
- **Polling interval:** 10 minutes
- **Auth:** None
- **Dedup key:** `source=GDACS, external_id=item.guid or item.link`
- **Parsing:** XML with `feedparser` or `xml.etree.ElementTree`. Extract `geo:lat` and `geo:long` from GeoRSS namespace.
- **Field mapping:**
  | GDACS Field | Schema Field |
  |-------------|-------------|
  | `item.guid` or hash of `item.link` | `external_id` |
  | `"GDACS"` | `source` |
  | parsed from `gdacs:eventtype` or title keywords | `type` |
  | `item.title` | `title` |
  | `item.description` (strip HTML tags) | `description` |
  | `gdacs:alertlevel` → GREEN=LOW, ORANGE=MODERATE, RED=HIGH | `severity` |
  | `"ACTIVE"` | `status` |
  | `geo:lat` | `lat` |
  | `geo:long` | `lon` |
  | `gdacs:bbox` if present → JSON | `geometry` |
  | parsed from `gdacs:country` or title | `country` |
  | null | `region` |
  | `item.pubDate` parsed | `started_at` |
- **GDACS eventtype → DisasterType:**
  - `EQ` → EARTHQUAKE
  - `TC` → STORM
  - `FL` → FLOOD
  - `VO` → VOLCANO
  - `WF` → WILDFIRE
  - `DR` → DROUGHT
  - else → OTHER

### OpenFEMA

- **Endpoint:** `https://www.fema.gov/api/open/v2/DisasterDeclarationsSummaries?$orderby=declarationDate desc&$top=100`
- **Polling interval:** 60 minutes
- **Auth:** None
- **Dedup key:** `source=FEMA, external_id=disasterNumber`
- **Field mapping:**
  | FEMA Field | Schema Field |
  |------------|-------------|
  | `disasterNumber` (as string) | `external_id` |
  | `"FEMA"` | `source` |
  | `incidentType` mapped (see below) | `type` |
  | `declarationTitle` | `title` |
  | `designatedArea` + `state` | `description` |
  | null (not granular enough) | `severity` |
  | `closeoutDate ? "CLOSED" : "ACTIVE"` | `status` |
  | geocoded from state centroid (hardcoded lookup) | `lat`, `lon` |
  | null | `geometry` |
  | `"United States"` | `country` |
  | `state` | `region` |
  | `declarationDate` parsed | `started_at` |
- **FEMA incidentType → DisasterType:**
  - `Fire`, `Coastal Storm`, `Tornado`, `Hurricane`, `Typhoon` → mapped accordingly
  - `Earthquake` → EARTHQUAKE
  - `Flood` → FLOOD
  - `Fire` → WILDFIRE
  - `Hurricane`, `Tornado`, `Severe Storm(s)`, `Coastal Storm`, `Typhoon` → STORM
  - `Volcanic Eruption` → VOLCANO
  - `Drought` → DROUGHT
  - else → OTHER
- **US State centroids:** Hardcoded dict of ~56 US state/territory name → (lat, lon). Include in `normalizer.py`.
- **Spatial jitter — required:** Because FEMA uses state centroids, multiple simultaneous active disasters in the same state will be placed at the exact same 3D coordinate on the globe. This causes Z-fighting (flickering) in Three.js's InstancedMesh depth sorting. The Python worker must apply a small, deterministic random offset to FEMA coordinates: `lat += (hash(external_id) % 100 - 50) / 100.0 * 0.5` and `lon += (hash(external_id + "lon") % 100 - 50) / 100.0 * 0.5`. This spreads events within a ±0.25° radius (~28km) of the state centroid — visually tight enough to still indicate the state, but far enough apart on the 3D sphere to eliminate Z-fighting. Use Python's built-in `hash()` seeded on the external_id for determinism (same event always gets the same jitter).

### NASA FIRMS

- **Endpoint:** `https://firms.modaps.eosdis.nasa.gov/api/area/csv/{MAP_KEY}/VIIRS_SNPP_NRT/world/1`
  - Returns last 24h of global fire detections as CSV
- **Polling interval:** 30 minutes
- **Auth:** MAP_KEY env var. **Skip entirely if not set.**
- **Dedup key:** `source=FIRMS, external_id=geohash_of_cluster_center_at_precision_4`
  - FIRMS has no unique ID. Do **not** use a composite of exact lat/lon/date/time — the same fire will produce different detections across polls and days, creating duplicate events for a single ongoing fire.
  - Instead, group detections into ~10km grid cells using **Geohash at precision 4** (~40km × 20km per cell). Compute the Geohash of each detection point; group all detections sharing the same Geohash prefix (precision 4). Each unique Geohash cell = one event. The `external_id` is `"FIRMS_{geohash4}"` (e.g. `"FIRMS_9q5f"`).
  - On each poll: if a record for that Geohash already exists in the DB, **update** its `updated_at`, `status` (keep ACTIVE), and `description` (update confidence/satellite info). Do **not** create a new event. This ensures one persistent event per active fire cluster regardless of how many polls occur.
  - Install `python-geohash==0.8.5` and add to `requirements.txt`.
- **Note:** FIRMS returns thousands of points. After grouping by Geohash precision 4, compute each cluster's centroid as the mean lat/lon of all detections in the cell. Title: "Active Fire — {geohash4} ({centroid lat:.2f}, {centroid lon:.2f})".
- **Field mapping:**
  | FIRMS Field | Schema Field |
  |-------------|-------------|
  | `"FIRMS_{geohash4}"` | `external_id` |
  | `"FIRMS"` | `source` |
  | `"WILDFIRE"` | `type` |
  | `"Active Fire — {lat}, {lon}"` | `title` |
  | `"Satellite: {satellite}, Confidence: {confidence}"` | `description` |
  | confidence-based: low<30→LOW, 30-70→MODERATE, 70-90→HIGH, >90→EXTREME | `severity` |
  | `"ACTIVE"` | `status` |
  | cluster centroid | `lat`, `lon` |
  | cluster bbox | `geometry` |
  | null | `country` |
  | null | `region` |
  | `acq_date + acq_time` | `started_at` |

### NOAA/NWS Alerts

- **Endpoint:** `https://api.weather.gov/alerts/active?status=actual`
- **Polling interval:** 10 minutes
- **Auth:** None. Set `User-Agent` header to `"DisasterPlatform/1.0 (contact@example.com)"` as required by NWS API.
- **Dedup key:** `source=NOAA, external_id=features[].properties.id`
- **Field mapping:**
  | NWS Field | Schema Field |
  |-----------|-------------|
  | `properties.id` | `external_id` |
  | `"NOAA"` | `source` |
  | `properties.event` mapped (see below) | `type` |
  | `properties.headline` | `title` |
  | `properties.description` (truncate to 2000 chars) | `description` |
  | `properties.severity` mapped | `severity` |
  | `properties.status == "Actual" && !properties.expires past? "ACTIVE" : "CLOSED"` | `status` |
  | centroid of `geometry.coordinates` or first point of affected zone | `lat`, `lon` |
  | full GeoJSON geometry if polygon | `geometry` |
  | `"United States"` | `country` |
  | `properties.areaDesc` | `region` |
  | `properties.onset` or `properties.sent` | `started_at` |
- **NWS severity → Severity:**
  - `Minor` → LOW, `Moderate` → MODERATE, `Severe` → HIGH, `Extreme` → EXTREME
- **NWS event → DisasterType:**
  - Contains "Tornado", "Hurricane", "Storm", "Wind", "Thunder" → STORM
  - Contains "Flood", "Flash Flood" → FLOOD
  - Contains "Fire", "Red Flag" → WILDFIRE
  - Contains "Volcano" → VOLCANO
  - Contains "Earthquake" → EARTHQUAKE
  - Contains "Drought" → DROUGHT
  - else → OTHER

### ReliefWeb

- **Endpoint:** `https://api.reliefweb.int/v1/reports?appname=disaster-platform&limit=50&preset=latest&profile=list`
  - Add filter: `fields[include][]=title&fields[include][]=date&fields[include][]=country&fields[include][]=disaster_type&fields[include][]=url`
- **Polling interval:** 60 minutes
- **Auth:** None
- **Purpose:** Primarily enrichment. Creates events only for reports tagged with a specific disaster type and country.
- **Dedup key:** `source=RELIEFWEB, external_id=report.id`
- **Field mapping:**
  | ReliefWeb Field | Schema Field |
  |-----------------|-------------|
  | `report.id` (as string) | `external_id` |
  | `"RELIEFWEB"` | `source` |
  | `disaster_type[0].name` mapped | `type` |
  | `report.title` | `title` |
  | null | `description` |
  | null | `severity` |
  | `"ACTIVE"` | `status` |
  | country centroid from hardcoded lookup | `lat`, `lon` |
  | null | `geometry` |
  | `country[0].name` | `country` |
  | null | `region` |
  | `date.created` parsed | `started_at` |

### Change Detection & EventUpdate Records

When an upsert modifies an existing event, the worker:
1. Loads the current DB row for that (source, external_id) before update
2. Compares tracked fields: title, description, severity, status, lat, lon, geometry, country, region
3. If any differ, creates an EventUpdate record with:
   - `changed_fields`: JSON array of field names that changed (e.g. `["severity", "status"]`)
   - `snapshot`: JSON of the full event state BEFORE the update
4. Then performs the UPDATE

This is done in a single transaction per event to avoid race conditions.

## 10. ARCHITECTURE.md CONTENT

```markdown
# Architecture — Global Disaster Intelligence Platform

## System Overview

A real-time geospatial intelligence platform that ingests, normalizes, enriches, and visualizes global disaster data. Three independently deployed services share a single PostgreSQL database.

## Data Flow

```
┌──────────────────────────────────────────────────────────────────────┐
│                        EXTERNAL DATA SOURCES                         │
│  USGS · EONET · GDACS · FEMA · FIRMS · NOAA · ReliefWeb             │
└──────────────┬───────────────────────────────────────────────────────┘
               │ HTTP polling (5–60 min intervals)
               ▼
┌──────────────────────────┐
│   INGESTION WORKER       │  Python · FastAPI · APScheduler
│   (Railway)              │  - One ingestor per source
│                          │  - Normalizes to unified schema
│   GET /health            │  - Deduplicates on (source, external_id)
│                          │  - Logs field changes as EventUpdates
└──────────┬───────────────┘
           │ PostgreSQL INSERT/UPDATE
           ▼
┌──────────────────────────┐
│   NEON POSTGRESQL        │  Shared database
│                          │  Tables: events, event_updates, source_status
│                          │  Accessed by both worker (writes) and app (reads)
└──────────┬───────────────┘
           │ Prisma Client queries
           ▼
┌──────────────────────────┐         ┌─────────────────────┐
│   NEXT.JS APPLICATION    │         │  ENRICHMENT APIs     │
│   (Vercel)               │◄───────►│  - Gemini Flash 2.0  │
│                          │         │  - Open-Meteo         │
│   /api/* — REST API      │         │  - ReliefWeb          │
│   / — 3D Globe (Three.js)│         └─────────────────────┘
│   /events/:id — Detail   │
│     (SSR + Leaflet)      │
└──────────────────────────┘
           │
           ▼
       [ Browser ]
```

## Deployment Topology

| Service          | Platform | Tier   | Purpose                      |
|------------------|----------|--------|------------------------------|
| Next.js App      | Vercel   | Free   | Frontend + API routes        |
| PostgreSQL       | Neon     | Free   | Persistent storage           |
| Ingestion Worker | Railway  | $5/mo  | Background data polling      |

## Database Schema

Three tables:
- **events** — Unified disaster event records from all sources. Composite unique on (source, external_id) for deduplication.
- **event_updates** — Changelog entries created when an existing event's tracked fields change during re-ingestion.
- **source_status** — One row per data source tracking last fetch time, event count, health, and errors.

## API Layer

All endpoints are Next.js API routes under `/api/`:
- `GET /api/events` — Filterable event list (type, severity, source, bbox, since)
- `GET /api/events/:id` — Full event detail
- `GET /api/events/:id/updates` — Change history
- `POST /api/events/:id/summarize` — On-demand AI summary generation
- `GET /api/sources` — Ingestion health dashboard
- `GET /api/stats` — Aggregated counts for dashboard

## AI Integration Points

### Current (MVP)
- **Situation Summaries**: Gemini Flash 2.0 generates 3-sentence summaries per event on first page visit. Cached in DB after generation. Rate: ~15 RPM, 1500/day free.

### Future (V2)
- **Anomaly Detection**: Compare incoming events against historical baselines. Flag unusual patterns (e.g., earthquake swarm in a normally quiet zone). Integration point: post-normalization hook in the ingestion pipeline.
- **Trend Narration**: Daily/weekly auto-generated reports summarizing disaster trends by region. Integration point: scheduled job in the worker + new API route.
- **Cross-Source Correlation**: Use LLM to determine if events from different sources describe the same incident. Integration point: `event_group_id` field + new correlation worker job.
- **Impact Assessment**: Estimate affected population using event geometry + population density data. Integration point: `impact_score` and `population_exposure` fields.

## V2 Extension Points

These are designed into the architecture but not implemented:

| Feature                  | Structural Groundwork                                    |
|--------------------------|----------------------------------------------------------|
| Real-time push (SSE)     | `/api/events/stream` stub returns 501                    |
| Cross-source grouping    | `event_group_id` field in events table                   |
| Impact scoring           | `impact_score` + `population_exposure` fields            |
| Satellite imagery        | `satellite_image_url` field                              |
| News enrichment          | Add `news_articles` JSON field in future migration       |
| WebSocket live updates   | Replace SSE stub with WebSocket or SSE implementation    |

## Known Tradeoffs

1. **Polling vs. push**: All data sources are polled. This means events appear with up to N-minute delay (where N is the polling interval for that source). Acceptable for MVP — most sources don't offer push anyway.
2. **Python + Node split**: Two runtimes add deployment complexity but each excels at its job. The Python worker handles XML parsing, CSV processing, and scheduled polling. Node handles SSR, API routing, and React rendering.
3. **On-demand AI summaries**: Summaries are generated when users visit event pages, not proactively. This conserves the 1500/day free tier quota but means the first visitor sees a loading state.
4. **FIRMS clustering**: Fire detections are clustered by proximity (0.1°) to avoid thousands of individual point events. This loses some granularity but keeps the event count manageable.
5. **FEMA geocoding**: FEMA data only includes state names, not coordinates. We use hardcoded state centroids, which places markers at state centers rather than exact disaster locations. Acceptable for the map overview; the detail page notes the actual designated area.
6. **Single database**: Both services hit the same Neon instance. At scale, you'd want read replicas or a dedicated analytics store. For free-tier traffic, this is fine.

## Future Migration Notes

- **Adding a new data source**: Create a new ingestor file in `worker/ingestors/`, add the source to the `EventSource` enum (requires a Prisma migration), register the job in `scheduler.py`.
- **Switching to real-time**: Implement the SSE endpoint (`/api/events/stream`). The worker would need to notify the Next.js app of new events — options: database triggers + pg_notify, or a shared Redis pub/sub channel.
- **Adding user accounts**: New Prisma models for User, SavedFilter, AlertSubscription. Auth via NextAuth.js. This is a net-new feature, not a refactor.
- **Schema sync risk — Prisma vs. Python raw SQL (CRITICAL):** The Python worker writes to the database using raw SQL column names (e.g. `external_id`, `started_at`). Prisma uses camelCase field names in the schema that map to snake_case column names via `@map()`. If a Prisma field is renamed, the `@map()` column name may also change — silently breaking the Python worker with a `psycopg2.errors.UndefinedColumn` at runtime, not at deploy time. **Mitigation:** (1) Never rename a `@map()` column name without updating `db.py` and all ingestors in the same commit. (2) The Python worker must run a column existence check at startup: `SELECT column_name FROM information_schema.columns WHERE table_name='events'` and assert the expected set of snake_case columns are all present. If any column is missing, log a `FATAL` error and exit — this surfaces the mismatch immediately rather than silently inserting partial data. Implement this check in `db.py`'s `startup_check()` function, called from `main.py` before the scheduler starts.
```
